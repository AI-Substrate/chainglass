## Purpose

Agents must call `cg wf node` CLI commands to interact with the orchestrator. Today the prompt hardcodes `cg` which resolves to a globally-linked binary that can point at the wrong branch, the wrong repo snapshot, or not exist at all. This workshop designs a system where short, self-contained shell scripts are placed inside the workspace before any pod runs, so agents always use the correct CLI version — the one from the same codebase they're running in.

## Key Questions Addressed

- Where should shortcut scripts live within the workspace?
- What commands need shortcuts and what do they look like?
- Which service creates them (SRP) and who orchestrates that?
- How does the agent prompt reference them?
- What happens in test vs production?

## Problem Statement

```
Global cg shim → /substrate/030-.../apps/cli/dist/cli.cjs   (STALE)
                                     ↑
In-process code → /substrate/033-.../packages/...            (CURRENT)
                                     ↑
                              Schema mismatch → E157
```

The globally-linked `cg` is a cross-cutting tool that any branch can relink. When the agent spawns `cg wf node accept ...` it gets whichever version was last linked globally. This is fragile, untestable, and invisible when it breaks.

**Principle**: The agent should use CLI commands that are **co-versioned** with the orchestrator that spawned it.

## Design

### Location: `.chainglass/bin/`

Shortcut scripts live at `<worktreePath>/.chainglass/bin/` inside the workspace.

```
<workspace>/
└── .chainglass/
    ├── bin/                          ← NEW: shortcut scripts
    │   ├── cg-accept
    │   ├── cg-collate
    │   ├── cg-get-input
    │   ├── cg-save-output
    │   ├── cg-end
    │   ├── cg-ask
    │   ├── cg-error
    │   └── cg-get-answer
    ├── data/
    │   └── workflows/
    ├── graphs/
    └── units/
```

**Why `.chainglass/bin/`**:

| Option | Pros | Cons |
|--------|------|------|
| `<workspace>/.chainglass/bin/` | Co-located with all other WF state. Already in `.chainglass` which is the orchestration data root. Cleaned up naturally with workspace. | One extra path segment |
| `<workspace>/bin/` | Short path | Pollutes workspace root. Conflicts with user's own `bin/`. |
| `<workspace>/.cg/` | Short | New hidden dir convention. Not discoverable. |
| Injected into `$PATH` | Agent uses bare `cg` | Fragile. Process env manipulation. |

**Decision**: `.chainglass/bin/` — consistent with existing `.chainglass/` conventions, no workspace pollution.

### Script Design

Each script is a thin wrapper that calls `node` with the correct CLI entrypoint, forwarding arguments. Scripts are self-contained — no dependency on PATH, global installs, or symlinks.

#### Exemplar: `cg-accept`

```bash
#!/usr/bin/env bash
# Auto-generated by WorkflowBinService — do not edit
# Accepts the agent's assignment on a workflow node.
# Usage: .chainglass/bin/cg-accept <graphSlug> <nodeId>
set -euo pipefail
CLI_ENTRY="%%CLI_ENTRY%%"
exec node "$CLI_ENTRY" wf node accept "$@"
```

`%%CLI_ENTRY%%` is resolved at generation time to the absolute path of the current repo's compiled CLI entrypoint (e.g., `/home/jak/substrate/033-real-agent-pods/apps/cli/dist/cli.cjs`).

#### Exemplar: `cg-save-output`

```bash
#!/usr/bin/env bash
# Auto-generated by WorkflowBinService — do not edit
# Saves output data for a workflow node.
# Usage: .chainglass/bin/cg-save-output <graphSlug> <nodeId> <outputName> '<jsonValue>'
set -euo pipefail
CLI_ENTRY="%%CLI_ENTRY%%"
exec node "$CLI_ENTRY" wf node save-output-data "$@"
```

#### Full Command Map

| Script | Resolves To | Agent Usage |
|--------|-------------|-------------|
| `cg-accept` | `cg wf node accept` | `.chainglass/bin/cg-accept {{graphSlug}} {{nodeId}}` |
| `cg-collate` | `cg wf node collate` | `.chainglass/bin/cg-collate {{graphSlug}} {{nodeId}}` |
| `cg-get-input` | `cg wf node get-input-data` | `.chainglass/bin/cg-get-input {{graphSlug}} {{nodeId}} <name>` |
| `cg-save-output` | `cg wf node save-output-data` | `.chainglass/bin/cg-save-output {{graphSlug}} {{nodeId}} <name> '<val>'` |
| `cg-end` | `cg wf node end` | `.chainglass/bin/cg-end {{graphSlug}} {{nodeId}} --message "..."` |
| `cg-ask` | `cg wf node ask` | `.chainglass/bin/cg-ask {{graphSlug}} {{nodeId}} --type text --text "..."` |
| `cg-error` | `cg wf node error` | `.chainglass/bin/cg-error {{graphSlug}} {{nodeId}} --code X --message "..."` |
| `cg-get-answer` | `cg wf node get-answer` | `.chainglass/bin/cg-get-answer {{graphSlug}} {{nodeId}} <qId>` |

**Why individual scripts instead of one `cg` wrapper?**
- Agent sees named commands → less prompt token overhead per invocation
- Agent can't accidentally call non-WF commands (e.g., `cg workspace remove`)
- Each script has its own usage comment → agent self-documents if confused
- Scripts are short (~5 lines each) so total footprint is small

### WorkflowBinService (New Service — SRP)

A single-responsibility service that generates and manages the `.chainglass/bin/` scripts.

```typescript
/**
 * WorkflowBinService: Generates version-locked CLI shortcut scripts.
 *
 * Ensures agents always use the same CLI version as the orchestrator.
 * Scripts are idempotent — calling ensure() multiple times is safe.
 */
export interface IWorkflowBinService {
  /**
   * Ensure all shortcut scripts exist and are current.
   * Creates .chainglass/bin/ if missing, writes/updates scripts.
   * Idempotent — safe to call before every pod execution.
   */
  ensure(ctx: { readonly worktreePath: string }): Promise<void>;

  /**
   * Get the absolute path to the bin directory.
   */
  getBinDir(ctx: { readonly worktreePath: string }): string;
}
```

#### Implementation Sketch

```typescript
import { resolve, dirname } from 'node:path';
import { fileURLToPath } from 'node:url';

const COMMANDS: Array<{ name: string; args: string; description: string }> = [
  { name: 'cg-accept',      args: 'wf node accept',          description: 'Accepts the agent assignment' },
  { name: 'cg-collate',     args: 'wf node collate',         description: 'Collates all node inputs' },
  { name: 'cg-get-input',   args: 'wf node get-input-data',  description: 'Reads an input by name' },
  { name: 'cg-save-output', args: 'wf node save-output-data', description: 'Saves output data' },
  { name: 'cg-end',         args: 'wf node end',             description: 'Completes the node' },
  { name: 'cg-ask',         args: 'wf node ask',             description: 'Asks a question' },
  { name: 'cg-error',       args: 'wf node error',           description: 'Reports an error' },
  { name: 'cg-get-answer',  args: 'wf node get-answer',      description: 'Reads answer to a question' },
];

export class WorkflowBinService implements IWorkflowBinService {
  private readonly cliEntry: string;

  constructor() {
    // Resolve CLI entrypoint relative to this package
    // apps/cli/dist/cli.cjs from the repo root
    this.cliEntry = resolveCliEntry();
  }

  async ensure(ctx: { readonly worktreePath: string }): Promise<void> {
    const binDir = this.getBinDir(ctx);
    await fs.mkdir(binDir, { recursive: true });

    for (const cmd of COMMANDS) {
      const scriptPath = join(binDir, cmd.name);
      const content = this.generateScript(cmd);
      await fs.writeFile(scriptPath, content, { mode: 0o755 });
    }
  }

  getBinDir(ctx: { readonly worktreePath: string }): string {
    return join(ctx.worktreePath, '.chainglass', 'bin');
  }

  private generateScript(cmd: typeof COMMANDS[number]): string {
    return [
      '#!/usr/bin/env bash',
      '# Auto-generated by WorkflowBinService — do not edit',
      `# ${cmd.description}`,
      'set -euo pipefail',
      `exec node "${this.cliEntry}" ${cmd.args} "$@"`,
      '',
    ].join('\n');
  }
}
```

#### Resolving the CLI Entrypoint

The service needs to find `apps/cli/dist/cli.cjs` from the running process. Two options:

**Option A — Relative from package**: The service lives in `packages/positional-graph/`. Walk up to repo root and resolve `apps/cli/dist/cli.cjs`.

```typescript
function resolveCliEntry(): string {
  // __dirname of this file → repo root → apps/cli/dist/cli.cjs
  const thisDir = dirname(fileURLToPath(import.meta.url));
  // Walk up from packages/positional-graph/dist/features/... to repo root
  const repoRoot = resolve(thisDir, '..', '..', '..', '..', '..');
  return resolve(repoRoot, 'apps', 'cli', 'dist', 'cli.cjs');
}
```

**Option B — Injected via DI/config**: The CLI entrypoint path is provided at construction time by the DI container which knows the repo layout.

**Recommendation**: Option A with a fallback to `which cg` for production deployments where the repo structure may differ. Test harness always uses Option A.

### Orchestration: Who Calls `ensure()`?

The `ensure()` call must happen **before any pod executes**. Two candidate call sites:

#### Option 1: ODS calls ensure() before pod.execute()

```typescript
// In ODS.handleAgentOrCode()
await this.deps.workflowBinService.ensure({ worktreePath: ctx.worktreePath });
const pod = this.deps.podManager.createPod(nodeId, podParams);
pod.execute({ ... });
```

**Pros**: Single call site. Every pod gets bins.
**Cons**: ODS gains a new dependency. Runs for every node (idempotent but extra I/O).

#### Option 2: OrchestrationService calls ensure() once per drive()

```typescript
// In OrchestrationService or DriveHandle, at start of drive()
await this.deps.workflowBinService.ensure({ worktreePath: ctx.worktreePath });
// ... then proceed with iterations
```

**Pros**: Called once per drive session, not per-node. Cleaner separation.
**Cons**: If someone calls ODS directly without OrchestrationService, bins might be missing.

#### Option 3: PodManager calls ensure() at createPod() time

```typescript
// PodManager.createPod()
createPod(nodeId: string, params: PodCreateParams): IWorkUnitPod {
  // Bins are guaranteed by the time any pod exists
  this.binService.ensureSync(this.currentCtx);
  // ...
}
```

**Cons**: PodManager is an internal collaborator, not a good place for I/O orchestration. Violates SRP.

**Recommendation: Option 2** — OrchestrationService (or DriveHandle) calls `ensure()` once when a drive session starts. This is the natural "setup before execution" point. ODS stays focused on node dispatch.

### Prompt Template Changes

The starter and resume prompts update to use shortcut scripts:

```markdown
## Step 1: Accept Your Assignment

Before doing anything else, accept your assignment:

\`\`\`
.chainglass/bin/cg-accept {{graphSlug}} {{nodeId}}
\`\`\`

## Step 2: Read Your Task

\`\`\`
.chainglass/bin/cg-collate {{graphSlug}} {{nodeId}}
.chainglass/bin/cg-get-input {{graphSlug}} {{nodeId}} main-prompt
.chainglass/bin/cg-get-input {{graphSlug}} {{nodeId}} <inputName>
\`\`\`
```

All 8 `cg wf node ...` references change to `.chainglass/bin/cg-*` references.

**Note**: Since agent cwd is already set to `worktreePath`, relative paths like `.chainglass/bin/cg-accept` just work. No absolute paths needed in the prompt.

### Test Infrastructure

`withTestGraph` already creates `.chainglass/` subdirectories. Adding `bin/` is a one-line change:

```typescript
// In withTestGraph(), after creating directories:
const binService = new WorkflowBinService();
await binService.ensure({ worktreePath: tmpDir });
```

Or, if the real DriveHandle calls `ensure()`, the test's `orchestrationService.get(ctx, slug)` handles it automatically.

## Scope and Boundaries

### In Scope
- `WorkflowBinService` class (generate + ensure)
- 8 shortcut scripts
- Prompt template updates (starter + resume)
- `ensure()` call wired into OrchestrationService or DriveHandle
- Test infrastructure update (`withTestGraph`)

### Out of Scope
- Removing global `cg` link (still useful for human CLI use)
- Changing `cg` command structure or registration
- Any changes to the adapter layer (SdkCopilotAdapter, ClaudeCodeAdapter)
- PATH manipulation

## Open Questions

### Q1: Should scripts embed `--workspace-path` automatically?

**RESOLVED: No.** The agent's cwd IS the workspace root. `cg` resolves context from cwd via the workspace registry. Adding `--workspace-path` would add noise and create a second source of truth for workspace location.

### Q2: Should we generate a single `cg` wrapper instead of per-command scripts?

**RESOLVED: No, use per-command scripts.** Per-command scripts are safer (agent can't call arbitrary `cg` subcommands), more discoverable (each has a usage comment), and reduce prompt token usage per invocation. Total footprint is ~40 lines across 8 scripts.

### Q3: What if the CLI dist is stale (not built)?

**OPEN**: `resolveCliEntry()` could check if `cli.cjs` exists and warn/fail early. In practice, `just install` or `pnpm build` keeps dist current. For tests, the build is always run before test execution. Could add a guard:

```typescript
if (!existsSync(this.cliEntry)) {
  throw new Error(
    `CLI entrypoint not found: ${this.cliEntry}. Run 'pnpm build' first.`
  );
}
```

### Q4: Where does WorkflowBinService live (package location)?

**RESOLVED: `packages/positional-graph/src/features/030-orchestration/`** — it's an orchestration infrastructure concern, co-located with PodManager, ODS, and the prompt templates it serves. Not PlanPak'd because it's cross-cutting orchestration infrastructure.

## Purpose

Resolve how templates are created and how node identity works. The key insight: **templates are saved from working graph instances, not authored from scratch in a new YAML format.** This eliminates the need for a separate `workflow.yaml` declarative format — the existing `graph.yaml` + `nodes/*/node.yaml` files ARE the template definition.

## Key Questions Addressed

- How are templates created? (Answer: "Save as template" from a working graph instance)
- Do templates need a new declarative YAML format? (Answer: No — reuse existing graph.yaml + node.yaml)
- How do node IDs work in templates? (Answer: Already present — generated during original graph creation)
- What gets included in a template vs stripped out?
- How does instantiation work without an ID mapping step?

---

## The Insight: Templates Come From Instances

The original workshop (001) proposed a new `workflow.yaml` format — a declarative graph definition that would need a parser to convert it into imperative `service.create()`/`addNode()`/`setInput()` calls. This had a hidden complexity: node IDs are generated by the service, so the parser would need an ID mapping layer.

**The user's insight eliminates this entirely:**

```
CREATION FLOW:
1. User builds a graph imperatively (CLI: cg wf create, cg wf node add, cg wf node set-input)
2. Graph works — nodes have real IDs, wiring is validated, everything runs
3. User says "save this as a template" (CLI: cg template save-from <graph-slug>)
4. System copies graph definition files + bundles work units → template directory

INSTANTIATION FLOW:
1. User instantiates template (CLI: cg template instantiate <template> --id <instance>)
2. System copies graph.yaml + nodes/*/node.yaml + units/ → instance directory
3. System creates fresh state.json (graph_status: pending, empty nodes map)
4. Done — graph is ready to activate with EXISTING node IDs
```

**No new YAML format. No parser. No ID mapping. The existing positional graph persistence format IS the template format.**

## What a Template Actually Is

A template = a graph's **definition files** (topology + wiring) + **bundled work units** (self-contained dirs), with all runtime state stripped out.

### Files Included in Template (from graph instance)

| File | Source | Purpose | Template-Safe? |
|------|--------|---------|----------------|
| `graph.yaml` | `.chainglass/data/workflows/<slug>/graph.yaml` | Lines, node IDs, orchestrator settings | ✅ Yes — pure topology |
| `nodes/<id>/node.yaml` | `.chainglass/data/workflows/<slug>/nodes/<id>/node.yaml` | unit_slug, input wiring, per-node orchestrator settings | ✅ Yes — pure config |

### Files EXCLUDED from Template (runtime artifacts)

| File | Why Excluded |
|------|-------------|
| `state.json` | Runtime execution state — statuses, timestamps, events, Q&A |
| `nodes/<id>/outputs/` | Agent-generated output data |
| `nodes/<id>/data/` | Session IDs, input caches |
| `nodes/<id>/events.json` | Node event log |

### Work Units Bundled Alongside

| File | Source | Purpose |
|------|--------|---------|
| `units/<slug>/unit.yaml` | `.chainglass/units/<slug>/unit.yaml` | Unit definition |
| `units/<slug>/prompts/main.md` | `.chainglass/units/<slug>/prompts/main.md` | Agent prompt template |
| `units/<slug>/scripts/main.sh` | `.chainglass/units/<slug>/scripts/main.sh` | Code unit script |

## Operations (Revised)

### Save as Template

```
$ cg template save-from advanced-pipeline --as my-pipeline-template

┌──────────────────────────────────────────────────────────────┐
│ STEP 1: Read graph definition                                │
│   Source: .chainglass/data/workflows/advanced-pipeline/       │
│   • Copy graph.yaml (topology)                               │
│   • Copy nodes/*/node.yaml (wiring — NOT outputs/data/state) │
│   • Extract unique unit_slugs from all node.yaml files       │
└──────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────┐
│ STEP 2: Bundle work units                                    │
│   For each unique unit_slug:                                 │
│   • Copy .chainglass/units/<slug>/ → template/units/<slug>/  │
│   • Include unit.yaml, prompts/, scripts/                    │
│   • chmod +x all .sh files                                   │
└──────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────┐
│ OUTPUT                                                       │
│   Target: .chainglass/templates/workflows/my-pipeline-template/
│                                                              │
│   ✓ Template saved: my-pipeline-template                     │
│   ✓ Graph: 4 lines, 6 nodes                                 │
│   ✓ Units bundled: human-input, spec-writer, programmer-a,   │
│     programmer-b, reviewer, summariser                       │
│   ✓ Runtime state excluded (state.json, outputs, events)     │
└──────────────────────────────────────────────────────────────┘
```

### Instantiate

```
$ cg template instantiate my-pipeline-template --id sprint-42

┌──────────────────────────────────────────────────────────────┐
│ STEP 1: Copy template → instance                             │
│   • Copy graph.yaml → instances/<slug>/<id>/graph.yaml       │
│   • Copy nodes/*/node.yaml → instances/<slug>/<id>/nodes/    │
│   • Copy units/ → instances/<slug>/<id>/units/               │
│   • Write instance.yaml (metadata)                           │
└──────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────┐
│ STEP 2: Initialize runtime state                             │
│   • Create data/instances/<slug>/<id>/state.json             │
│     { graph_status: "pending", nodes: {}, questions: [] }    │
│   • Copy graph.yaml + nodes/ to data path for engine         │
│   • Graph is immediately ready for orchestration             │
└──────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────┐
│ OUTPUT                                                       │
│                                                              │
│   ✓ Instance created: my-pipeline-template/sprint-42         │
│   ✓ Graph ready with 4 lines, 6 nodes (node IDs preserved)  │
│   ✓ 6 units copied locally                                  │
│   ✓ Fresh state.json initialized                             │
└──────────────────────────────────────────────────────────────┘
```

**Key difference from Workshop 001**: No separate "activate" step. Instantiation copies the graph definition to both the Git-tracked instance dir AND the runtime data dir. The graph is immediately runnable.

## What This Eliminates

| Workshop 001 Concept | Status | Why |
|---------------------|--------|-----|
| `workflow.yaml` (new declarative format) | **ELIMINATED** | graph.yaml IS the declarative format — already exists |
| workflow.yaml parser (Phase 3 task 3.1-3.2) | **ELIMINATED** | No new format to parse |
| `GraphActivationService` (Phase 3 task 3.3-3.4) | **SIMPLIFIED** | Becomes a file copy + fresh state.json, not a YAML→imperative bridge |
| Node ID mapping (`logicalId → physicalId`) | **ELIMINATED** | IDs already exist in the template from when the graph was originally built |
| `cg wf activate` as separate step | **MERGED** into instantiate | Instantiation includes graph setup |

## Revised Instance Directory Structure

```
.chainglass/instances/<template-slug>/<instance-id>/
├── instance.yaml                       ← Metadata (template ref, created_at, units manifest)
├── graph.yaml                          ← Copied from template (topology with real node IDs)
├── nodes/
│   ├── human-input-a1b/
│   │   └── node.yaml                  ← Copied from template (wiring)
│   ├── spec-writer-c3d/
│   │   └── node.yaml
│   └── ...
└── units/                              ← Copied from template (self-contained)
    ├── human-input/
    │   └── unit.yaml
    ├── spec-writer/
    │   ├── unit.yaml
    │   └── prompts/main.md
    └── ...

.chainglass/data/instances/<template-slug>/<instance-id>/
├── graph.yaml                          ← Copied from instance (for engine to read)
├── state.json                          ← Fresh: { graph_status: "pending", nodes: {} }
└── nodes/
    ├── human-input-a1b/
    │   └── node.yaml                  ← Copied from instance
    └── ...                            ← outputs/, data/, events created during execution
```

**Git tracking**:
- `.chainglass/instances/` → **tracked** (definitions + units)
- `.chainglass/data/instances/` → **gitignored** (runtime state, can be regenerated from instance)

## Impact on Plan Phases

### Phase 1: Schema & Interfaces (UPDATED)
- **TemplateValidationSchema** → NOT a new format. It's a **validation schema for template directories** — checks that graph.yaml exists, all node.yaml files are present, units/ directory has all referenced slugs.
- **No workflow.yaml Zod schema needed** — we validate graph.yaml using the existing `PositionalGraphDefinitionSchema`.

### Phase 2: Service + CLI (UPDATED)
- Add `cg template save-from <graph-slug> --as <template-slug>` — the primary template creation path
- `instantiate` now also initializes the graph runtime data (merge old Phase 3 into Phase 2)
- `refresh` operates on instance units/ only (graph topology doesn't change)

### Phase 3: Graph Activation (GREATLY SIMPLIFIED)
- ~~Complex parser~~ → Simple file copy from instance to data path + fresh state.json
- ~~ID mapping~~ → Not needed
- Essentially merges into Phase 2's instantiate step
- Phase 3 could focus on **integration testing + edge cases** instead

### Phase 4: E2E Migration (UPDATED)
- `withTestGraph()` migration is simpler — build graph imperatively, save as template, then test instantiation
- Existing `buildAdvancedPipeline()` becomes the template creation script, not something to replace

## Open Questions

### Q1: Can templates be created from scratch (not from a graph)?

**RECOMMENDATION**: Not in Plan 048. The primary creation flow is "build graph → save as template." A future plan could add `cg template create --blank` that scaffolds an empty template, but it's not needed now.

### Q2: What happens to the original graph after save-from?

**RECOMMENDATION**: Nothing — the original graph continues to exist and run. Template save is a copy operation, not a move. The user can delete the original graph separately if desired.

### Q3: Should instantiate regenerate node IDs?

**RECOMMENDATION**: No. Preserving node IDs from the template means:
- Multiple instances of the same template have the same node IDs
- This is fine — they're in separate data directories with separate state.json
- Preserving IDs simplifies comparison between instances
- If uniqueness is ever needed, it's a future extension

## Quick Reference

```
TEMPLATE CREATION:
  graph instance → cg template save-from → template directory
  (copies: graph.yaml, nodes/*/node.yaml, units/)
  (strips: state.json, outputs/, data/, events)

INSTANTIATION:
  template → cg template instantiate → instance + runtime data
  (copies everything to instance dir)
  (copies graph.yaml + nodes/ to data dir)
  (creates fresh state.json)

REFRESH:
  template units/ → cg template refresh → instance units/ overwritten
  (graph topology unchanged, only units refreshed)
```

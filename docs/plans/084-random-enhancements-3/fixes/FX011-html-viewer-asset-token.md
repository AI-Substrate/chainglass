# Fix FX011: HtmlViewer asset access ŌĆö short-lived signed token

**Created**: 2026-05-16
**Status**: Proposed
**Plan**: [auth-bootstrap-code-plan.md](../auth-bootstrap-code-plan.md) (the same plan that introduced the cookie gate)
**Source**: User-reported during file-browser usage on `higgs-jordo` workspace ŌĆö `storyboard.html` renders but every embedded `<img>` returns `401 {"error":"bootstrap-required"}` (verified via curl). HtmlViewer was added by commit `9266b45b` (pre-Plan-084); when Plan 084 landed the always-on bootstrap cookie gate, asset loading inside the sandboxed iframe silently broke because `sandbox="allow-scripts"` gives the iframe an opaque origin and the browser strips the HttpOnly cookie from every request the iframe makes.
**Domain(s)**: `_platform/auth` (modify ŌĆö additive contract); `file-browser` (modify ŌĆö HtmlViewer consumes the new contract)

---

## Problem

HtmlViewer renders HTML files inside a sandboxed iframe (`sandbox="allow-scripts"`, deliberately no `allow-same-origin` per Plan 084's security posture). The iframe's opaque origin means the browser does not send the `chainglass-bootstrap` HttpOnly cookie when the iframe fetches `<img>`, `<link>`, or `<script>` references ŌĆö every asset request hits the proxy with no cookie and is rejected with `401 bootstrap-required`. The HTML body itself loads because the *parent* React component fetches it (parent origin, cookie attached) and hands the result to the iframe via a blob URL ŌĆö but the iframe is on its own for every sub-resource.

This regressed when Plan 084 Phase 3 introduced the always-on cookie gate; before that, the raw-file route was gated only by Auth.js (which had different cookie scoping). The fix has to thread an alternate credential through the URLs that end up inside the iframe ŌĆö and the credential cannot be the cookie itself (JS can't read HttpOnly cookies).

The implication is **not** that the sandbox should be relaxed: doing so would let any HTML the user opens drive the full app API as the authenticated user. Plan 084 spent six phases hardening this surface. The fix must preserve the sandbox.

## Proposed Fix

Introduce a short-lived (10 min), HMAC-signed **asset token** scoped to a specific worktree, derived from the same signing key the bootstrap cookie uses. The parent React (which holds the cookie) calls a new mint endpoint, receives a token, and HtmlViewer splices it into every rewritten asset URL as `&_at=<token>`. The proxy and raw-file route both accept this token as alternate auth for the raw-file route only.

Three layers change:

1. **`@chainglass/shared/auth-bootstrap-code`** ŌĆö add stateless `buildAssetToken(worktree, key, ttl)` and `verifyAssetToken(token, worktree, key, now)` mirroring the existing `buildCookieValue`/`verifyCookieValue` HMAC pattern.
2. **Web app** ŌĆö add `POST /api/bootstrap/asset-token` (cookie-gated) that mints `{ token, expiresAt }`; teach `bootstrapCookieStage` to short-circuit to `'bypass'` for `/api/workspaces/<slug>/files/raw` when `?_at=<token>` is valid; teach the raw route handler to accept the same token in lieu of `await auth()`.
3. **HtmlViewer** ŌĆö on mount, POST the mint endpoint with the worktree extracted from `src`; append `&_at=<token>` to each rewritten asset URL.

### Key Reuse Justification (NIST SP 800-108 type-tagging)

The asset token reuses the bootstrap signing key but with a **distinct HMAC input prefix** (`"asset:"` vs the bare bootstrap code that the cookie HMAC consumes). This is a standard domain-separation pattern: different prefixes produce disjoint HMAC outputs even under the same key, so an attacker cannot trick a verifier into accepting a cookie HMAC as an asset-token HMAC or vice versa. Rotating the bootstrap key invalidates both cookies and outstanding asset tokens, which is the correct shared-fate semantics (one rotation, one revocation event). No new key material is introduced.

### Authoritative Validation Layer

Two layers (proxy + raw-route handler) both check the token. The **proxy is authoritative** for unauthenticated callers ŌĆö its short-circuit returns `'bypass'` (which skips Auth.js) on valid token, or falls through to the cookie gate on invalid token. The **handler check is defense-in-depth** for the edge case where a request reaches the handler because cookie was valid but bears an `_at` query param: the handler must distinguish "token present and valid ŌåÆ skip auth", "token present and INVALID ŌåÆ reject (don't fall back)", and "token absent ŌåÆ call auth()". This explicit three-branch shape closes a subtle gap under `DISABLE_GITHUB_OAUTH=true` where `auth()` returns a fake session ŌĆö invalid tokens must reject explicitly rather than fall through.

### Critical Assumptions (verify during FX011-1 first)

1. `getBootstrapCodeAndKey()` returns `{ code, key }` where `key` is the same `Buffer` already used for cookie HMAC (verified by inspection of `apps/web/src/lib/bootstrap-code.ts` and `packages/shared/src/auth-bootstrap-code/cookie.ts`).
2. The Next.js route `app/api/workspaces/[slug]/files/raw/route.ts` produces pathname `/api/workspaces/<slug>/files/raw` exactly ŌĆö no trailing slash, no rewriting.
3. The HtmlViewer's `src` URL reliably carries `?worktree=<absolute-path>` (confirmed by reading `file-viewer-panel.tsx:632` where `rawFileUrl` is constructed).
4. The `chainglass-bootstrap` cookie is sufficient for the parent React's fetch to the mint endpoint to succeed (the parent shares the app origin).
